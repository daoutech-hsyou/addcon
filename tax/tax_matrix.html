4. 현금영수증 발급 API 이용 기준 (개인, 즉시/주문완료 발행)

문서 단위로 taxationType은 과세 또는 비과세 하나만 지정. 필드 구성: supplyCost(공급가액) + tax(부가세) + serviceFee(봉사료) = totalAmount(거래금액).

5. 우리가 구현해야 할 발행 (요약)

  • 현금영수증 발행 — 개인(선불) + [충전잔액 · 가상계좌 · 전용계좌] (과세=과세, 면세=비과세)
  • 세금계산서 발행 — 기업(선불) 과세분 + [충전잔액 · 가상계좌 · 전용계좌], 기업(후불) 월정산 과세분
  • 계산서(면세) 발행 — 기업(선불/후불) 면세분 (세금계산서와 별도 건)
  • 구현 제외(결제사 발행) — 신용카드 · 휴대폰결제 · 네이버/카카오페이

6. 구현 체크리스트

  • 회원·결제 마스터: 개인/기업선불/기업후불, 기업 사업자번호 진위확인, 후불 여신·미수금
  • 상품 마스터: 과세 / 면세 / 쿠폰(상품권성) 구분 플래그 — 모든 발행 로직의 출발점
  • 선수금(충전) 원장: 충전=선수금 무발행, 사용=차감, 충전 카드전표는 매출 배제
  • 증빙 결정 룰엔진: (상품유형 × 회원 × 결제수단) → (증빙종류 · 발행주체 · 시점) 판정, 위 매트릭스 데이터화
  • 혼합주문 금액분리: 과세공급가액/과세부가세/면세금액 분리 집계
  • 발행 연동: 현금영수증·세금계산서·계산서 (PG 또는 ASP[팝빌 등] — 세금계산서/계산서까지면 ASP 통합 권장)
  • 발행 상태관리: 대기/완료/실패 + 승인번호 저장 + 실패 재시도, 기명발급 기본값
  • 이중발급 차단: 카드/간편결제 건은 우리 발행 대상에서 자동 제외 플래그
  • 취소·부분취소: (-)수정세금계산서/계산서, 현금영수증 취소, 선수금 복원 (과세·면세 각각)
  • 거래명세서: 주문내역 발급 유지 + "세무증빙 아님" 표기, 발행 증빙번호 병행 노출
  • 결제사 발행 수단(카드·휴대폰·간편결제)의 발급 주체·방식 연동 스펙 확인
  • 매출 ↔ 증빙 발급 대사 리포트(누락·중복 탐지)

7. 세무 확정 필요 (개발 착수 전)

  • 쿠폰의 세법상 성격 (상품권성 vs 물품·용역형) 및 회사 역할 (직접판매 vs 중개)
  • 기업 즉시 카드결제 시 세금계산서 발급 허용 여부 (원칙적으로 카드결제분은 세금계산서 발급 제한)
  • 현금영수증/세금계산서 의무발행업종 해당 여부 (2026년 의무발행업종 확대)
  • 혼합주문 현금영수증 분리발행 vs 단일발행 처리 방침
